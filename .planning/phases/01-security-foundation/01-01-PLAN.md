<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> JWT Filter | Token may be malformed, expired, tampered, or absent |
| Free-text input -> Backend | User-supplied text may contain XSS payloads |
| Client -> API endpoint | Token length abuse could cause parsing overhead |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | JwtUtil.verify() | mitigate | Pin RSASSA/ECDSA verification (already in place), reject tokens over 2048 chars before parsing, generic "Invalid token" for all failures |
| T-01-02 | Denial of Service | JwtAuthenticationFilter | mitigate | Token length limit prevents parsing overhead on oversized payloads |
| T-01-03 | Information Disclosure | JwtAuthenticationFilter catch block | mitigate | Return RFC 7807 with generic "Invalid or expired token" — no algorithm, expiry, or signature details leaked |
| T-01-04 | Spoofing | UserProfile seeding | mitigate | Admin seed uses ON CONFLICT for idempotent application; role column constrained to ('admin','user') |
| T-01-05 | Tampering | Free-text input fields | mitigate | InputSanitizer strips XSS payloads as defense-in-depth; primary protection is no user-generated HTML rendering |
</threat_model>
